MySQL底层原理：chapter1基础篇

发表于 2024-03-29 更新于 2024-04-01 分类于 MySQL ， 底层原理

全文参考：

mysql执行流程

两层：Server层+存储引擎层

  • Server层：建立连接、解析sql语句、执行sql语句
  • 存储引擎层：数据的存储和提取。不同存储引擎可共享一个Server层。

两层之间以记录为最小单位传输数据。

连接器

  • 流程：TCP三次握手建立连接 -> 用户密码输入 -> 用户权限获取
  • 查询连接情况：show processlist查看多少个客户端连接了mysql
  • 最大连接数：show variables like 'max_connections'
  • 最长空闲连接时长：show variables like 'wait_timeout'定义了最长的空闲连接时长，超出则断开连接。（28880s=8h）
  • 长连接和短链接（和HTTP类似）
    • 短连接，每个sql语句执行需要TCP3握，执行完毕TCP4挥。
    • 长连接，一个TCP连接可以执行多个sql语句。但会导致内存占用增多。
      • 解决方式1：服务器定期断开长连接
      • 解决方式2：客户端主动重置长连接。虽然重置了，但不需要再重连和重新验证权限

查询sql缓存（废弃）

先查缓存，以sql语句为key，结果为value。对于更新频繁的表命中率低，因此8.0版本后移除。

该缓存与buffer pool不同。

词法分析 -> 语法分析，构建语法树。此过程不会查看表或字段是否存在。

执行sql

预处理

  • 确定sql语句中的表/字段是否存在
  • 将select *的*扩展为全列

优化器

基于查询成本，确认sql的执行计划。例如在多种索引时确定用哪个索引。

执行器

执行器开始与存储引擎交互，

  • 主键索引查询：计划中用到了主键索引，优化器决定访问类型为const。先查B+树，有记录时判断记录是否符合条件，符合发给客户端，否则跳过。
  • 全表扫描：计划中没有用到索引，优化器决定访问类型为ALL进行查询。循环所有记录，符合条件发给客户端。
  • 索引下推：能够减少【二级索引】查询时的回表操作。

索引下推？

5.6版本推出的机制，减少『二级索引』在查询时的『回表操作』。通常和联合索引有关？

叫『下推』的原因是，一些应该是Server层处理的事情，可『下推』到存储引擎层处理。

mysql数据存储方式

一张数据库表的数据是保存在名为『表名字.ibd』 的『独占表空间文件』里。

此处主要讲InnoDB的结构，表空间由『段、区、页、行』（segment，extent，page，row）组成

各结构概览

行 row

记录按行存放，不同的存储引擎有不同结构。

详见后续行结构细节

页 page

问题：每次I/O只读取一行，效率很差。

解决：多条记录存为一页，一次的I/O以页为单位。即当读取一条记录时，会以页为单位从磁盘读到内存。

页是存储引擎管理的最小单位，为16KB，即最多能保证一页16KB的连续空间

区 extent

问题：InnoDB 存储引擎用 B+ 树存储数据，每个叶子通过双向链表链接，如果相邻两页距离很远，『对磁盘而言』需要大量的『随机I/O』，速度较慢。

解决：多个相邻的页的物理位置也相邻，构成区，因此可以使用『顺序I/O』，范围查询时的性能很高。数据大时，为索引分配空间是按照区，而不是按照页为单位分配。

区大小1MB = 64 × 页大小16KB

段 segment

段由多个区组成，表空间则由多个段构成。段分为

  • 索引段：存放B+树非叶子节点的区
  • 数据段：存放B+树叶子节点的区
  • 回滚段：存放了回滚数据的区

行格式细节

行格式即存储记录的结构，InnoDB提供了4种格式：

  • Redundant格式用于5.0之前，基本不使用。
  • Compact格式，紧凑行格式，5.0后。
  • Dynamic和Compressed，也是紧凑行格式，类似Compact。

本节以Compact介绍，可通过Compact拓展到其他紧凑行格式。

具体而言，除了真实数据本身，Compact每一行记录的额外信息有：变长字段长度列表，NULL值列表、记录头信息、真实数据部分的隐藏字段。

变长字段长度列表

变长字段有：varchar，TEXT，BLOB等。

『变长字段长度列表』根据建表时『变长字段的逆序』存放『变长字段』的数据真正占用的字节数。

注意，『变长字段长度列表』中各元素的长度必须以字节为单位，由于mysql对记录大小的限制，因此每个元素最大为2字节。当表里没有变长字段时，该列表可以不存在。

NULL值列表

某些列可能有NULL值，将NULL值存为真实数据浪费空间。

『NULL值列表』根据建表时『允许NULL字段的逆序』存放表示字段是否为NULL的二进制位：0时为NULL；1时相反。

注意，NULL值列表长度必须以字节为单位，位数不足时『高位补0』。类似的，如果所有字段NOT NULL，则该列表可以不存在。

记录头信息

内容较多，比较重要的是

  • delete_mask：软删除的数据标记为1
  • next_record：下一条记录的位置，实际上指的是下一条记录的『记录头信息』和『真实数据』之间的位置。因此往右可以读到记录头信息，往左可以读到真实数据。
  • record_type：0表示普通记录，1表示B+树非叶子节点记录，2表示最小记录，3表示最大记录。

隐藏字段

除了建表时定义的字段，还有一些隐藏字段。这些字段属于数据部分。

  • row_id：建表时没有指定主键，且表里没有UNIQUE约束的列时（即表内没有字段可以作为主键），会被作为隐藏主键使用，因此是非必需的。占用6字节。
  • trx_id：指定该记录由哪个事务生成。必需，占用6字节。
  • roll_pointer：指定上一个版本的指针。必需，占用7字节。

varchar(n)的n最大取值

mysql规定，除了TEXT和BLOBs大对象类型，其他所有列（不包括『隐藏列』和『记录头信息』，包括『变长字段长度列表』和『NULL值列表』）占用的字节长度不超过65535字节。

即满足式子：

所有用户定义的字段长度（除了TEXT和BLOBs字段） + 变长字段长度列表所占字节数 + NULL值列表所占字节数 <= 65535

例如下表，只创建一个varchar字段，那么n最大可取多大？

  • 『变长字段长度列表』所占字节数：2字节。因为varchar字段的长度应该是接近65535字节，因此需要2字节记录varchar字段长度。
  • 『NULL值列表』所占字节数：1字节。因为可以只用1bit表示是否为null，且该列表要以字节为单位。

因此，根据上式 n + 2 + 1 <= 65535，即n最大取65532。

行溢出处理方式

一页大小为16KB=16384字节，小于65535字节，因此一些记录会发生『行溢出』。

  • Compact：在记录真实数据的地方保存『该列的部分数据』，并将剩余数据放到『溢出页』，在部分数据后用20字节指针，指向该『溢出页』。
  • Compressed和Dynamic：不存部分数据，直接将数据放到『溢出页』，只存20字节的指针。

Go标准库math包

注意：参与math包运算的输入与结果全为float64！

1、绝对值与取整

  • Abs(x float64) float64：返回x的绝对值
  • Ceil(x float64) float64：返回不小于x的最小整数值。
  • Floor(x float64) float64：返回不大于x的最大整数值。
  • Round(x float64) float64：返回四舍五入到最接近的整数值。

Ceil And Floor In Python W3schools | Viewfloor.co (frameimage.org)

对于负值而言，五入是向下入，相当于floor，即入了后数值更小；正数则是向上入，相当于ceil，即入了后数值更大。

2、次方

  • Pow(x, y float64) float64：返回x的y次幂。
  • Sqrt(x float64) float64：返回x的平方根。
  • Exp(x float64) float64：返回自然指数e的x次幂。

3、对数

  • Log(x float64) float64：返回x的自然对数。
  • math.Ln2：2的自然对数。
  • math.Log2E：以2为底的自然对数e的倒数。
  • math.Log10E：以10为底的自然对数e的倒数。

4、三角函数

  • math.Pi：圆周率π。
  • Sin(x float64) float64：返回x的正弦值（x以弧度为单位）。
  • Cos(x float64) float64：返回x的余弦值（x以弧度为单位）。
  • Tan(x float64) float64：返回x的正切值（x以弧度为单位）。

5、大小比较

  • Max(x, y float64) float64
  • Min(x, y float64) float64

Go标准库bufio包

bufio为io操作提供了缓冲区（内存），降低访问磁盘次数，以提高文件读写的效率。例如，有多次写入同一个文件时，可以先多次写到内存，再一次性写入磁盘，减少io系统调用次数。

bufio封装io.Writer和io.Reader接口的对象，并返回一个也实现该接口的对象

  • 写时要注意Flush的使用

使用bufio读

bufio.Reader

bufio.NewReaderSize

  • 将io.Reader对象rd封装为缓存大小为size的*bufio.Reader对象
  • size小于16时会被设置为16

bufio.NewReader

  • 创建size为4096默认大小的*bufio.Reader对象。

reader.Read

为bufio.Reader对象可调用方法

  • 从b中读取长度为len(p)的内容，存放到p中
  • 情况1：当缓冲区buf有内容时，将缓冲区内容全部填入p并刷新缓冲
  • 情况2：当缓冲区buf没内容时
    • len(p)>len(buf)，即读取的内容大小>缓冲大小，直接读文件。
    • len(p)<len(buf)，即读取的内容大小<缓冲大小，文件内容被读到缓冲中。随后，将缓冲中长度len(p)的数据读到p中，此时缓冲有剩余len(buf)-len(p)内容，可供下次读取（即情况1）。

这意味着，在情况2中，若读取的内容比较大时，bufio会将部分内容读进缓冲（预读），后续进行多次从缓冲中读取即可，直到读空缓冲，缓冲再装填。

reader.Peek

为bufio.Reader对象可调用方法

Peek会返回缓存中前n个字节，但不会像reader.Read那样消费缓存中的数据。

  • 若n>len(buf)，即读取内容比缓存小，返回bufio.ErrBufferFull
  • 若n<=len(buf)，
    • 若peek中不包含EOF，返回n个字节。
    • 若peek中还包含EOF，返回EOF前的所有字节，error会返回EOF。

reader.ReadSlice

bufio.Reader对象可调用方法

从b的缓冲区中读取数据到line，直到遇到第一个byte类型delim。读取的过程会消耗缓存。

  • 情况1：在缓存内遇到了delim，返回line和nil
  • 情况2：读完缓存了，没遇到EOF，也没遇到delim，返回line和EOF错误
  • 情况3：遇到了EOF都没遇到delim，返回line和io.ErrBufferFull错误

注意：

ReadSlice返回的是原始缓存中的内容，如果针对缓存作并发操作，则返回的内容有可能被其他操作覆盖。因此在官方注释里面有写，建议使用ReadBytes或ReadString。但ReadBytes和ReadString涉及内存申请和拷贝，因此会影响性能。在追求高性能的场景下，建议外部使用sync.pool来提供缓存。

reader.ReadBytes

与ReadSlice类似，不同之处在于

  1. 返回[]byte是新的切片，适用于高并发时，而ReadSlice返回原始缓存的切片（众所周知，原始切片被修改后，从其切出来的切片也会被修改）。
  2. 会从缓存中一直读，缓存也一直刷新，直到找到了或EOF

reader.ReadLine

  • 从缓存读取一行，底层使用了ReadSlice('\n')，并移除该行\n或\r\n。
  • 如果没有遇到换行符，也不会EOF或io.ErrBufferFull，而是将isPrefix置为真。

reader.Buffered

bufio.Reader对象可调用方法

返回b的缓冲区的剩余的字节数。

使用bufio写

bufio.Writer

  • 对bufio.Writer写入时，不再直接写入wr，而是先写入缓存buf，当缓存写满或使用了flush后再写入wr。减少了多次写。

bufio.NewWriterSize

  • 将io.Writer对象w封装为缓存大小为size的*bufio.Writer对象
  • size大于0即可

bufio.NewWriter

  • 将io.Writer对象w封装为缓存大小为4096的*bufio.Writer对象

writer.Write

bufio.Writer对象可调用方法

  • 将p中内容向b中写入
  • 情况1：len(p)>len(buf)，即写入的内容大小>缓冲大小，直接写磁盘而不写入缓存。
  • 情况2：len(p)<len(buf)，即写入的内容大小<缓冲大小
    • 当缓冲区buf满时，将缓冲区内容内容写入磁盘
    • 当缓冲区buf仍有空间时，不会写入磁盘，除非调用Flush

writer.Buffered

为bufio.Writer对象可调用方法

返回b的缓冲区的已存入的字节数。

writer.Available

为bufio.Writer对象可调用方法

返回b的缓冲区的剩余可存入的字节数。

demo

  • 此demo打开了可写的文件，并转为了缓冲区大小为5的bufio.writer对象writer。
  • 第一次往writer中写入了长度为6的字符串，直接写入文件。因此Buffered为0。（情况1）
  • 第二次往writer中写入了长度为5的字符串，虽然缓冲容量也为5，但显然写入后，Buffered仍为5，即没有刷新，此时数据并未写入文件中。对应情况2中buf未满。

  • 若直接运行上述代码，最后4字节的字符串将会在缓冲中，由于缓冲未满，而不会被写入到文件（情况2中buf未满）。若取消注释，则调用了Flush()，才会被写入到文件。

时间概念：

  1. 秒（Second）： 秒是国际单位制中最基本的时间单位。
  2. 毫秒（Millisecond）： 一毫秒等于一秒的千分之一，即 0.001 秒。
  3. 微秒（Microsecond）： 一微秒等于一秒的百万分之一，即 0.000001 秒。
  4. 纳秒（Nanosecond）： 一纳秒等于一秒的十亿分之一，即 0.000000001 秒。

time包中几个重要的类型

时间time.Time

直接打印time.Time

2024-01-27 16:54:00.820804 +0800 CST m=+0.001517601

  1. 日期： “2024-01-27”
  2. 时间： “16:54:00”
  3. 毫秒： “.820804” 表示毫秒部分，即 0.820804秒。
  4. 时区： “+0800 CST” 表示时区为中国标准时间 (CST)，偏移量为 +08:00。

“m=+0.001517601” 是 Go 语言中的一个时间戳增量（timestamp increment）部分。它通常出现在 Go 的时间字符串中，表示时间相对于某个基准时间的增量。

在这个增量部分中：

  • “m=” 表示这是一个时间戳增量的标识。
  • “+0.001517601” 表示时间戳相对于某个基准时间增加了 0.001653801 秒。

因此，整个时间戳字符串表示的是在基准时间2024-01-27 16:54:00.820804 +0800 CST上增加了约 0.001517601 秒的时间点。这在某些应用中用于测量事件的时间差或计时。

问题：json解析的时间格式处理

Go标准库的json包使用RFC3339标准定义的时间格式：

RFC3339格式：“{年}-{月}-{日}T{时}:{分}:{秒}.{毫秒}{时区}”；
其中的年要用零补齐为4位，月日时分秒则补齐为2位。毫秒部分是可选的。
最后一部分是时区，例如 Z 其实是零时区 Zulu 的缩写，它也可能是 +08:00 或 -08:00 等；
2017-12-08T00:00:00.00Z
2017-12-08T08:00:00.00+08:00
都代表所在时区的本地时间。

因此json通常无法反序列化常用的时间字符串。而在序列化时也只能得到RFC3339标准的字符串。

Fyne——基于go的GUI库

发表于 2023-12-27 分类于 Golang ， 非标准库包

Fyne

简单介绍

一种基于Go的GUI。Golang的GUI没接触过，也有挺多库可用，选择Fyne的原因是star比较多，但Fyne的中文文档并不全面。本篇文章不会完整介绍Fyne的所有组件，主要是总结一下使用Fyne过程中对该库的一些个人理解。

fyne-io/fyne: Cross platform GUI toolkit in Go inspired by Material Design (github.com)

快速开始 | Fyne 开发文档 (go-circle.cn)

Fyne的安装

需求：

  • golang（已满足）
  • C编译器
  • 显卡驱动（已满足）

C编译器官方文档给出了3种安装方式，但不论哪种都需要配置环境变量。否则会在编译Fyne库时会报错：

我选择的是TDM-GCC Download | tdm-gcc (jmeubank.github.io)。

环境变量：安装时记住安装的路径，例如C:\TDM-GCC-64，并复制gcc.exe所在所在路径，即C:\TDM-GCC-64\bin，最后在设置环境变量时，在系统变量Path中新增一项bin文件的路径即可。

几个关键概念

  • 窗口Window，略。
  • 画布Canvas，一个窗口有一个画布。
  • 画布对象CanvasObject，可用被放到画布上展示的元素。
  • 容器Container，一次存放多个画布对象的画布对象。
  • 布局Layout，规定了容器内的各个画布对象的尺寸与位置。

几个关键接口概览

CanvasObject接口

CanvasObject接口描述了一个可以被加到画布canvas的对象。

Layout接口

Layout描述了许多CanvasObject怎么样以特定的尺寸被摆放。

当一个CanvasObject同时实现了Layout接口时，Resize()和Move()方法将会失效。

Fyne中绘制的一切都是CanvasObject，因此几乎所有的结构体都实现了CanvasObject接口。例如下面的常见结构体。

  • 绘图或动画
    • Rectangle 长方形
    • Text 文本
    • Circle 圆圈
    • Image 图像
  • 组件Widget或WidgetRenderer，用于和用户交互的主要部分。
    • Label 标签
    • Button 按钮
    • Entry 输入框
  • 容器Container，用于以一定的布局Layout包括一系列的CanvasObject，而容器本身也实现了CanvasObject接口。
    • Box 盒子
    • Grid 流式布局
    • GridWrap 弹性流式布局
    • Border 边界布局

布局和容器

container和layout的关系

通俗来讲，容器是放元素的，而布局规定了容器内的元素该如何被摆放。一定程度上，可以认为布局就是容器

Container用于将多种CanvasObject按一定的Layout布局或不按照布局排列拜访。同时Container本身也是CanvasObject，这意味着容器内也可以放容器。

container有两种实例化的方法，其区别就是是否带布局Layout。

  • 如果是New方法，其内部的元素尺寸和位置将由Layout自动决定。
  • 如果是NewWithoutLayout方法，其内部的元素尺寸和位置将由手动决定（手动较为复杂，通常用自动布局）。

Fyne提供的布局

Fyne提供了很多种布局（容器），各布局有其元素尺寸和位置的摆放规则。

布局列表 | Fyne 开发文档 (go-circle.cn)

通过使用多种布局，可以构建更复杂的应用程序结构。

Border边界布局demo

我们只需要通过一个布局demo即可理解布局是如何影响各画布元素的尺寸和位置。

Border布局创建方法：

输入的CanvasObject会先后按照如下原则放置（建议跳过直接看demo进行理解）：

  • 前两个元素分别布局在上下，高度=元素的最小高度，宽度=窗口的最大宽度；
  • 随后两个元素分别布局在左右，宽度=元素的最小宽度，高度=窗口最大高度-上下元素高度之和；
  • 后面的多个CanvasObject将会按顺序先后以这些元素的最大尺寸填充到中间区域，这个区域被包围在前四个元素内的区域，且稍小一些。

其中关注NewBorder所包含的元素，参照放置原则：

  • 前两个分别是黑色文本和按钮，均以其最小的高度放了窗口的上下区域；
  • 随后两个分别是一个黑色长方形和输入框，均以其最小的宽度放了窗口的左右区域。值得注意的是，黑色长方形的最小宽度只有一个像素；
  • 最后三个，分别以黑色长方形，灰色圆形，白色文本分别以最大的尺寸先后放置在了中心区域。其中文本框已经是最大尺寸，只是文本比较小，且默认上下居中。
